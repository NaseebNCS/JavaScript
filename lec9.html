What is Object-Oriented Programming (OOP)?

Object-Oriented Programming (OOP) is a programming paradigm that is centered around objects rather than functions. Unlike some programming tools and frameworks that come and go, OOP remains relevant today because it is not a specific programming language or tool it is a style of programming, and a paradigm used across multiple languages.

Several programming languages support OOP, including C#, Java, Ruby, Python, and JavaScript. However, JavaScript’s object-oriented nature is somewhat controversial because it is primarily prototype-based rather than class-based, like Java or C#. Nevertheless, modern JavaScript (ES6 and beyond) introduced class syntax to support OOP concepts more explicitly

Understanding OOP is essential for serious developers, as it is a common topic in technical interviews and widely used in frameworks and libraries, such as Angular.

Key Concepts of OOP in JavaScript

ProtoTypes

In JavaScript, every object has a hidden property called a prototype.
👉 This prototype is like a backup object that provides properties and methods if they are not found in the original object.

Simple Idea Think of it like this:

  • You have a student object.
  • If the student doesn’t have a method like study(),
  • JavaScript will look into its prototype to find it.

This is called the prototype chain.

How to Access Prototype? You can access it using:

object.__proto__

(Though in modern JS, Object.getPrototypeOf() is preferred.)

Why Do We Use Prototype? 👉 To avoid repeating code (code duplication).
If many objects need the same method, we don’t define it again and again. Instead, we put it in the prototype, and all objects share it.

🏦 Example: Bank Account
❌ Without Prototype (Bad - duplication) 

function BankAccount(name, balance) {
  this.name = name;
  this.balance = balance;

  this.deposit = function(amount) {
    this.balance += amount;
  };
}

👉 Problem: Every object gets its own copy of deposit() → wastes memory.

With Prototype (Good - shared method)

function BankAccount(name, balance) {
  this.name = name;
  this.balance = balance;
}

BankAccount.prototype.deposit = function(amount) {
  this.balance += amount;
};

// Usage
let acc1 = new BankAccount("Naseeb Khan", 1000);
let acc2 = new BankAccount("Sara", 2000);

acc1.deposit(500);
console.log(acc1.balance); // 1500

👉 Both acc1 and acc2 share the same deposit method from prototype.

Student Example Using Prototype

Step 1: Create a Constructor Function

function Student(name, age) {
  this.name = name;
  this.age = age;
}

❌ Step 2: (Wrong Way – duplication)

function Student(name, age) {
  this.name = name;
  this.age = age;

  this.study = function() {
    console.log(this.name + " is studying");
  };
}

🚫 Problem: Every student gets its own copy of study() → wastes memory

✅ Step 3: Use Prototype (Correct Way)

Student.prototype.study = function() {
  console.log(this.name + " is studying");
};

👉 Now the method is shared by all students

Step 4: Create Objects

let s1 = new Student("Ali", 20);
let s2 = new Student("Sara", 22);

s1.study(); // Ali is studying
s2.study(); // Sara is studying

👉 Both objects are using the same method from prototype

How It Works (Simple)

When you do:

s1.study();

JavaScript checks:

  1. Does s1 have study()? ❌ No
  2. Go to Student.prototype ✅ Found
  3. Run it

Add More Methods

You can add more shared methods:

Student.prototype.play = function() {
  console.log(this.name + " is playing");
};

Add More Properties

Student.prototype.school = "NaseebCodeStudio School"; // always use prototypes for methods specially

🔗 Prototype Chain

s1 → Student.prototype → Object.prototype → null

What is Prototype Chain?

If something is not found in an object:

  1. JS looks inside the object
  2. Then in its prototype
  3. Then prototype’s prototype
  4. And so on...

👉 This chain continues until it reaches null 

// Visual Idea
acc1 → BankAccount.prototype → Object.prototype → null

Important Points

  • Every object in JS has a prototype
  • Functions automatically get a prototype property
  • Methods added to prototype are shared
  • This helps save memory and improves performance

Prototype in JavaScript is a way to share properties and methods between objects without duplicating code.

  • Constructor → creates objects
  • Prototype → stores shared methods
  • Objects → use methods from prototype

The 4 Pillars of OOP

Object-Oriented Programming (OOP) is a programming style that helps organize code in a more structured and reusable way. Before OOP, procedural programming was used, where programs were divided into sets of functions. Data was stored in variables, and functions operated on that data. While this approach is simple, it can become messy as programs grow. You might end up with many functions scattered everywhere, copying and pasting code, and making changes in one place that break other parts of the program. This is called spaghetti code, and it becomes hard to maintain.

OOP solves this problem by grouping related variables and functions into units called objects. In OOP, variables are called properties, and functions are called methods. For example, a car can be an object with properties like make, model, and color, and methods like start(), stop(), and move(). Another example is the localStorage object in browsers, which has properties and methods like setItem() and removeItem().

The 4 pillars of OOP are:

  1. Encapsulation
  2. Abstraction
  3. Inheritance
  4. Polymorphism

Let’s focus on Encapsulation with examples.

Encapsulation

Encapsulation means grouping related variables (properties) and functions (methods) into an object. This makes the code easier to maintain and reduces the number of parameters in functions.

Example 1: Procedural Programming (Without OOP) In procedural programming, we use separate variables and functions. Here’s an example to calculate an employee’s wage: 

let baseSalary = 30_000;
let overtime = 10;
let rate = 20;

function getWage(baseSalary, overtime, rate) {
    return baseSalary + (overtime * rate);
}

console.log(getWage(baseSalary, overtime, rate)); // Output: 30,200

In this example, the getWage function has three parameters. If you need to change something, you have to pass these parameters every time, which can get messy.

Example 2: OOP with Encapsulatio In OOP, we group related variables and functions into an object. Here’s the same example using an object: 

let employee = {
 baseSalary: 30_000,
 overtime: 10,
 rate: 20,
 getWage: function() {
    return this.baseSalary + (this.overtime * this.rate);
 }
};

console.log(employee.getWage()); // Output: 30,200
In this example:
  • The employee object has three properties (baseSalary, overtime, and rate) and one method (getWage).
  • The getWage method has no parameters because it uses the object’s properties directly (this.baseSalary, this.overtime, and this.rate).

Why Encapsulation is Better

  1. Fewer Parameters: Functions in OOP often have fewer parameters because they use the object’s properties. As Robert C. Martin (Uncle Bob) says, “The best functions are those with no parameters.”
  2. Easier to Maintain: When related properties and methods are grouped together, it’s easier to understand and change the code.
  3. Less Spaghetti Code: Encapsulation helps avoid scattered functions and variables, making the code cleaner and more organized.

Abstraction

Think of a DVD player as an object. Inside, it has a complex logic board, but on the outside, it only has a few buttons. When you press a button, you don’t need to know what happens inside the DVD player. All that complexity is hidden from you. This is an abstraction in practice.

In JavaScript, we can use the same technique with objects. We can hide some properties and methods from the outside. This gives us two main benefits:

  1. Simpler Interface: The object becomes easier to use because you only see what’s necessary.
  2. Reduced Impact of Changes: If we change the inner, private methods, it won’t affect the outside code because no external code depends on those methods
class DVDPlayer {
 // Private method (hidden from the outside)
 #playDisc() {
 console.log("Playing the disc...");
 }

 // Public method (simple interface)
 play() {
 this.#playDisc();
 }
}

const player = new DVDPlayer();
player.play(); // Output: Playing the disc...
// player.#playDisc(); // This would cause an error because it's private

Inheritance

Inheritance is a mechanism that allows us to eliminate redundant code. For example, in HTML, elements like text boxes, checkboxes, and dropdowns have common properties (hidden, innerHTML) and methods (click(), focus()). Instead of redefining these for every element, we can define them once in a generic object (like HTMLElement) and let other objects inherit these properties and methods. 

class HTMLElement {
 constructor() {
 this.hidden = false;
 }

 click() {
 console.log("Element clicked!");
 }

 focus() {
 console.log("Element focused!");
 }
}

class TextBox extends HTMLElement {
 type() {
 console.log("Typing in the text box...");
 }
}

const textBox = new TextBox();
textBox.click(); // Output: Element clicked!
textBox.type(); // Output: Typing in the text box...

Polymorphism

"Poly" means many, and "morph" means form. So, polymorphism means "many forms." In object-oriented programming, polymorphism allows us to avoid long if-else or switch-case statements. For example, all HTML elements should have the ability to render themselves, but each element renders differently.

Without polymorphism, your code might look like this: 

switch (elementType) {
 case 'select':
    renderSelect();
    break;
 case 'text':
    renderTextBox();
    break;
 case 'checkbox':
    renderCheckbox();
    break;
 // ... and so on
}

With polymorphism, we can define a render() method for each object, and it will behave differently depending on the object type. This simplifies the code: 

class Element {
 render() {
    console.log("Rendering a generic element...");
 }
}
class Select extends Element {
 render() {
    console.log("Rendering a select box...");
 }
}
class TextBox extends Element {
 render() {
    console.log("Rendering a text box...");
 }
}

const elements = [new Select(), new TextBox()];
elements.forEach(element => element.render());

Output:
// Rendering a select box...// Rendering a text box...

Benefits of OOP

  1. Encapsulation: We group related variables and functions together, reducing complexity and making the code reusable.
  2. Abstraction: We hide unnecessary details and show only the essentials, reducing complexity and isolating the impact of changes.
  3. Inheritance: We eliminate redundant code by reusing properties and methods from a parent object.
  4. Polymorphism: We can refactor ugly switch-case or if-else statements into cleaner, more maintainable code.

By using these principles, your code becomes more organized, reusable, and easier to maintain.

    • Definition: The default or main execution context created when the script first runs.
    • Components
      • Global Object: In browsers, it’s the window object.
      • this Keyword: Refers to the global object in the global context.
      • Memory Allocation: Allocates memory for global variables and functions.
let globalVar = "I am global"; // Global variable
console.log(this); // Logs the global object (window in browsers)
  1. Functional Execution Context
    • Definition: Created every time a function is called. It’s like a mini-kitchen for that function.
    • Components
      • Local Variables: Variables declared inside the function.
      • Arguments: Parameters passed to the function.
      • this Keyword: Refers to the object calling the function (or the global object if not specified)
function greet(name) {
 let message = "Hello, " + name; // Local variable
 console.log(message);
}

greet("Naseeb"); // Calls the function, creates a new execution context
  1. Eval Execution Context
    • Definition: Created when the eval() function is used to execute code dynamically.
    • Note: Rarely used because it can be risky and inefficient.
let x = 10;
eval("let y = 20; console.log(x + y);"); // Logs 30
  1. Memory Creation Phase
    • Definition: The first phase of execution where memory is allocated for variables and functions.
    • Behavior:
      • Variables are initialized with undefined.
      • Functions are stored in memory as-is.
console.log(name); // undefined (memory allocated, but value not set yet)
var name = "Naseeb";
console.log(name); // "Naseeb" (value assigned during execution phase)
  1. Execution Phase
    • Definition: The second phase where the code is executed line by line.
    • Behavior:
      • Variables are assigned values.
      • Functions are executed.
let a = 5; // Memory allocated; value assigned
let b = 10;
console.log(a + b); // 15 (execution phase)
  1. New Execution Context
    • Definition: Created every time a function is called. It has its own memory and execution phase. 
function addNumbers(x, y) {
  let sum = x + y; // New execution context
  return sum;
}

let result = addNumbers(3, 4); // Call the function
console.log(result); // 7

Stack and Heap Memory?

In JavaScript, memory refers to the system's ability to store and manage data during program execution. Within memory, there are two main areas: the stack and the heap.

  1. Stack:
    • The stack is a region of memory that is used for static memory allocation, which means that memory allocation is determined at compile-time.
    • It is a sequential data structure that operates in a Last In, First Out (LIFO) manner.
    • The stack is primarily used for storing primitive data types, function calls, and local variables.
    • Memory allocation and deallocation on the stack are automatic and managed by the program execution.
  2. Heap:
    • The heap is a region of memory that is used for dynamic memory allocation, which means that memory allocation is determined at run-time.
    • Stores reference data type.
    • It is a more flexible and less organized memory structure compared to the stack.
    • The heap is primarily used for storing objects, arrays, and other complex data structures.
    • Memory allocation and deallocation on the heap are manual and need to be managed explicitly by the programmer.
    • Difference with an Example: Consider a scenario where we have a JavaScript function that creates an array and a local variable.

Code:

function createArray() { let localVariable = 10; let myArray = [1, 2, 3, 4, 5]; }

In this example:

  • The localVariable is stored on the stack because it is a primitive data type (number) and its memory allocation is determined at compile-time.
  • The myArray is stored on the heap because it is an array (an object) and its memory allocation is determined at run-time.

Here's a breakdown of memory usage:
Stack: Stores the memory address of the localVariable.
Heap: Stores the elements of the myArray (1, 2, 3, 4, 5) along with the memory addresses of each element.

LIFO

LIFO stands for Last In, First Out. It’s like a stack of plates:

  1. When you add a plate to the stack, you place it on top.
  2. When you remove a plate, you take it from the top.

The last plate you put on the stack is the first one you take out. That’s LIFO!

How does LIFO work in JavaScript?

In JavaScript, the Call Stack (a place where function calls are managed) follows the LIFO principle. Here’s how it works:

  1. When a function is called, it’s added to the top of the stack.
  2. When the function finishes, it’s removed from the top of the stack.

Example to Understand LIFO Let’s say you have three functions: first, second, and third. 

function first() {
 console.log("First function");
 second(); // Call the second function
}

function second() {
 console.log("Second function");
 third(); // Call the third function
}

function third() {
 console.log("Third function");
}

first(); // Start by calling the first function

What Happens in the Call Stack?

  1. Step 1: first() is called and added to the stack
    • Stack: [first]
  2. Step 2: Inside first(), second() is called and added to the stack.
    • Stack: [first, second]
  3. Step 3: Inside second(), third() is called and added to the stack.
    • Stack: [first, second, third]
  4. Step 4: third() finishes and is removed from the stack.
    • Stack: [first, second]
  5. Step 5: second() finishes and is removed from the stack.
    • Stack: [first]
  6. Step 6: first() finishes and is removed from the stack.
    • Stack: []

4 third() finishes [first, second]

use strict

In JavaScript, 'use strict' is a directive that enables strict mode, a stricter version of JavaScript. It helps you write cleaner, safer, and more efficient code by catching common mistakes and preventing the use of problematic features.

What is Strict Mode?

Strict mode is a way to opt into a restricted variant of JavaScript. It changes the behavior of the JavaScript engine in the following ways:

  1. Catches Silent Errors: Converts mistakes that JavaScript normally ignores into errors
  2. Prevents Accidental Global: Stops you from accidentally creating global variables.
  3. Disables Problematic Features: Disallows the use of deprecated or confusing features.
  4. Improves Performance: Helps the JavaScript engine optimize your code.

How to Enable Strict Mode

You can enable strict mode by adding the string 'use strict' at the beginning of a script or a function.

1. Global Strict Mode Add 'use strict' at the top of your script to enable strict mode for the entire file: 

'use strict';

Your code here 

x = 10; // This will throw an error because 'x' is not declared

2. Local Strict Mode Add 'use strict' inside a function to enable strict mode only for that function: 

function myFunction() {
    'use strict';

 // Strict mode applies only to this function
    y = 20; // This will throw an error because 'y' is not declared
}

Key Features and Benefits of Strict Mode

1. Prevents Accidental Global In non-strict mode, if you forget to declare a variable with var, let, or const, JavaScript automatically creates a global variable. Strict mode prevents this: 

'use strict';

x = 10; // Throws an error: "x is not defined"

2. Disallows Duplicate Parameter Names Strict mode prevents you from using duplicate parameter names in functions: 

'use strict';

function myFunction(a, a) { // Throws an error: "Duplicate parameter name not allowed"
 console.log(a);
}

3. Prevents Deleting Variables, Functions, or Arguments In strict mode, deleting variables, functions, or function arguments is not allowed: 

'use strict';

let x = 10;
delete x; // Throws an error: "Delete of an unqualified identifier in strict mode"

4. Makes this Safer In non-strict mode, if you call a function without specifying this, it defaults to the global object (window in browsers). In strict mode, this is undefined in such cases: 

'use strict';

function myFunction() {
 console.log(this); // Output: undefined (instead of the global object)
}

myFunction();

5. Disallows Octal Syntax In non-strict mode, numbers with a leading zero (e.g., 0123) are treated as octal (base 8). Strict mode disallows this: 

'use strict';

let x = 0123; // Throws an error: "Octal literals are not allowed in strict mode"

6. Prevents Writing to Read-Only Properties Strict mode throws an error if you try to modify a read-only property: 

'use strict';

const obj = {};
Object.defineProperty(obj, 'x', { value: 42, writable: false });

obj.x = 100; // Throws an error: "Cannot assign to read-only property 'x'"

7. Disallows with Statement The with statement is not allowed in strict mode because it can lead to confusing and unpredictable behavior: 

'use strict';

with (Math) { // Throws an error: "Strict mode code may not include a with statement"
 console.log(PI);
}

When to Use Strict Mode

  1. Always: It's a good practice to use strict mode in all your JavaScript code.
  2. Modern JavaScript: If you're using ES6+ (modern JavaScript), strict mode is automatically enabled for modules and classes.
  3. Legacy Code: If you're working with older code, enabling strict mode can help you identify and fix issues.

Example: Strict Mode in Action

Here’s an example that demonstrates the benefits of strict mode: 

'use strict';

// 1. Prevents accidental globals
x = 10; // Error: x is not defined

// 2. Disallows duplicate parameter names
function myFunction(a, a) { // Error: Duplicate parameter name not allowed
 console.log(a);
}

// 3. Prevents deleting variables
let y = 20;
delete y; // Error: Delete of an unqualified identifier in strict mode
// 4. Makes `this` safer
function myFunction() {
 console.log(this); // undefined
}
myFunction();

// 5. Disallows octal syntax
let z = 0123; // Error: Octal literals are not allowed in strict mode

// 6. Prevents writing to read-only properties
const obj = {};
Object.defineProperty(obj, 'x', { value: 42, writable: false });
obj.x = 100; // Error: Cannot assign to read-only property 'x'
