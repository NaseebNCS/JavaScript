Local Storage

If DOM is memory for the current page, then localStorage is memory for the browser.

Without localStorage:

Refresh Page
↓
Everything Lost 😭

With localStorage:

Refresh Page
↓
Data Still There 😎

This is why:

  • Todo apps use it
  • Pomodoro apps use it
  • Dark mode settings use it
  • User preferences use it

1. WHAT IS localStorage?

Think of localStorage as a small notebook 📒 inside the browser.

You can:

✅ Save data
✅ Read data
✅ Update data
✅ Delete data

And the data remains even after:

  • refresh
  • browser restart
  • computer restart

(until user clears it)

🎯 REAL LIFE EXAMPLE

Imagine a dark mode button.
Without localStorage: 

Dark Mode ON
↓
Refresh
↓
Back to Light Mode

Bad UX ❌
With localStorage: 

Dark Mode ON
↓
Refresh
↓
Still Dark Mode

Professional UX ✅

🧱 2. BASIC METHODS

There are only 4 methods you must master.

Method Purpose
setItem() Save
getItem() Read
removeItem() Delete one
clear() Delete all

💾 3. SAVING DATA

localStorage.setItem("username", "Naseeb"); // even after removing this line the storage still have the data

Think:

Key     → username
Value   → Naseeb

Like:

// JSON Format
{
  "username": "Naseeb"
}

📖 4. READING DATA

const user = localStorage.getItem("username");

console.log(user);

// Output: Naseeb

❌ 5. REMOVING DATA

localStorage.removeItem("username");

// Removes Only: username

🧹 6. CLEAR EVERYTHING

localStorage.clear();

Deletes ALL localStorage data.
Use carefully ⚠️

🎮 7. REAL PROJECT (Remember User Name)

🧠 9. IMPORTANT CONCEPT

localStorage stores ONLY STRINGS


This works:

localStorage.setItem("name", "Naseeb");

This also works:

localStorage.setItem("age", 25);

// But internally: "25" Stored as string.

⚠️ BIGGEST BEGINNER MISTAKE

Trying to store objects directly.

// ❌ Wrong
const user = {
  name: "Naseeb",
  age: 25
};

localStorage.setItem("user", user);

// Result: [object Object]

🔥 10. JSON.stringify()

Convert object → string

// JavaScript Code 
const user = {
  name: "Naseeb",
  age: 25
};

localStorage.setItem(
  "user",
  JSON.stringify(user)
);


// Stored as: JSON 
{
  "name": "Naseeb",
  "age": 25
}

🔥 11. JSON.parse()

Convert string → object

const user = JSON.parse(
  localStorage.getItem("user")
);

console.log(user.name);

// Output: Naseeb

🧠 GOLDEN RULE

// 1. Saving 
JSON.stringify()

// 2. Reading: 
JSON.parse()

🎮 12. REAL TODO APP STORAGE

const todos = [
  "Learn JS",

];

localStorage.setItem(
  "todos",
  JSON.stringify(todos)
);


// Load Todos

  localStorage.getItem("todos")
);

[
  "Learn JS",
  "Build Project"

⚡ 14. ENGINEERING BEST PRACTICES

// 1. ✅ Always check if data exists
const data = localStorage.getItem("user");

if (data) {
  // use it
}


// 2. ✅ Use constants for keys
// instead of: 
localStorage.setItem("user", data);
// Use:
const USER_KEY = "user"; // Large projects become easier to manage.


// 3. ✅ Store structured data
// Prefer:
{
  theme: "dark",
  username: "Naseeb"
} // Rather than dozens of separate keys. 


// 4. ✅ Parse once
// Avoid:
JSON.parse(...)
JSON.parse(...)
JSON.parse(...)
// repeatedly

⚠️ COMMON MISTAKES

// 1. ❌ Forgetting stringify
localStorage.setItem("user", user);


// 2. ❌ Forgetting parse
const user = localStorage.getItem("user"); // Now it's a string, not object.


// 3. ❌ Saving huge data, localStorage is small.
// Typically around: 5MB, Not for videos/images.


// 4. ❌ Storing passwords: Never do this.

🔥 STEP 11: EVENT DELEGATION

Instead of:

deleteBtn.addEventListener(...)

for every item,
Use bubbling. 

list.addEventListener("click", (e) => {

  if (e.target.tagName === "BUTTON") {

    const id =
      Number(
        e.target.parentElement.dataset.id
      );

    todos = todos.filter(todo => {

      return todo.id !== id;

    });

    saveTodos();

    renderTodos();

  }

});

🧠 ENGINEERING GOLD

This is Event Delegation, Used everywhere

Session Storage

Now that you understand localStorage, learning sessionStorage becomes very easy because they are almost twins.

The biggest difference is:

localStorage  = remembers forever
sessionStorage = remembers temporarily

🧠 1. WHAT IS sessionStorage?

Think of it as a temporary notebook 📒.
It stores data:
✅ While the tab is open
But when you close the tab:
❌ Everything is gone

🎯 REAL LIFE ANALOGY

Imagine you're solving an exam.
You write answers on rough paper. 

// While you're in the exam room:
Data exists

// When you leave:
Paper destroyed

That's sessionStorage.

🆚 2. localStorage VS sessionStorage

Feature localStorage sessionStorage
Refresh page ✅ survives ✅ survives
Close tab ✅ survives ❌ deleted
Close browser ✅ survives ❌ deleted
Storage size ~5MB ~5MB
Data type Strings Strings

🎮 EXAMPLE

Suppose user is filling a long form.
You don't want them to lose progress after refresh.
But you also don't want to permanently save it.

Perfect use case:

sessionStorage

🧱 3. SAVE DATA

Exactly like localStorage.

sessionStorage.setItem(
  "username",
  "Naseeb"
);

// Think:
// JSON code 
{
  "username": "Naseeb"
}

📖 4. READ DATA

  sessionStorage.getItem("username");

console.log(user);

// Output: Naseeb

❌ 5. DELETE ONE ITEM

  "username"
);

🧹 6. DELETE EVERYTHING

sessionStorage.clear();

Removes all session data.

🧠 7. OBJECTS & ARRAYS

Same rule as localStorage.

🍪 Cookies

Before learning authentication, JWT, sessions, or backend development, you should understand cookies.

Think of cookies as small notes that websites leave inside your browser.

🧠 What are Cookies?

Imagine you go to a restaurant.

The waiter gives you a token:

Table Number: 7

Next time you call the waiter, he knows:

"Oh, you're table 7."

Cookies work similarly.

A website gives your browser a small piece of information:

username = Naseeb
theme = dark
language = English

Then every time you visit that website again, the browser automatically sends those cookies back.

🎯 Why Cookies Exist

Cookies are used for:
✅ Login sessions
✅ Remember me feature
✅ User preferences
✅ Shopping carts
✅ Language settings
✅ Authentication

🧠 Cookie vs localStorage vs sessionStorage

Feature Cookies localStorage sessionStorage
Size ~4KB ~5MB ~5MB
Auto ent to server ✅Yes ❌No ❌No
Servives Refresh ✅Yes ✅Yes ✅Yes
Servives tab close Depends ✅Yes ❌No
Mainly used for Auth & sessions Persistent client data Temporary data

Creating Cookies

Cookies are stored using:

document.cookie

// Example:
document.cookie = "username=Naseeb";

// Browser Now stores:
username=Naseeb

Multiple Cookies

document.cookie = "theme=dark";
document.cookie = "language=en";

// Now Browser Contains:
username=Naseeb
theme=dark
language=en

Reading Cookies

console.log(document.cookie);

// Output:
username=Naseeb; 
theme=dark;
language=en

Notice: All cookies come back as one string.

Setting Expiration

Without expiration:

document.cookie = "username=Naseeb";

Cookie disappears when browser session ends.

Set Expiration:

document.cookie = "username=Naseeb; expires=Fri, 31 Dec 2030 23:59:59 UTC";

Now it survives browser restarts.

Path

document.cookie = "theme=dark; path=/";

This makes the cookie available throughout the whole website.

Updating Cookies

Simply overwrite:

document.cookie = "theme=light";

Deleting Cookies

Set Expiration to the past:

document.cookie = "theme=; expires=Thu, 05 Jan 1970 00:00:00 UTC";

Browser removes it.

Real Example: Remember User

Save:

document.cookie = "name=Naseeb";

// Read:
console.log(document.cookie);

Why Cookies Matter in Login Systems

Suppose user logs in.

// Server Creates:
sessionId = abc123xyz

// Browser stores:
sessionId=abc123xyz

// Whenever user visits another page:
Browser automatically sends: sessionId=abc123xyz

// Server recognizes:
"This is the same user."

// This is how many login systems work.

Cookies are Small

Cookies are only around: 4KB

Don't Store:
❌ Images
❌ Videos
❌ Large objects

document.cookie Is Not Very Friendly

Reading cookies gives:

username=Naseeb; theme=dark; language=en

So developers often create helper functions.

// Example:
// Set Cookie
function setCookie(name, value){
  document.cookie = `${name}=${value}; path=/`;
}

Get Cookie

function getCookie(name) {
  const cookies = document.cookie.split("; ");

  for (const cookie of cookies) {
    const [key, value] = cookie.split("=");

    if (key === name) {
      return value;
    }
  }

  return null;
}

// Usage:
setCookie("theme", "dark");

console.log(getCookie("theme"));

// Output: dark

EditThisCookie

We cannot directly see cookies inside the browser window. If you are running your website using Live Server, look at the address bar and click the icon located just to the left of your website URL.

  1. Click the icon beside the URL
  2. Select Manage Cookies and Site Data.
  3. Click Manage On-Device Site Data.
  4. You will see your website's data and a Delete Option, which allows you to remove stored cookies and site data.

However, if you want to view the actual cookie values, edit them, or delete specific cookies, install a browser extension called EditThisCookie.

With the EditThisCookie extension, you can:

  • View all cookies stored by your website.
  • Read cookies names and values.
  • Edit existing cookies.
  • Add new cookies.
  • Delete Individual cookies or clear all cookies.

This extension is very useful when testing and debuggin web applications that use cookies.

Think of cookies as small notes that websites save in your browser. The browser stores these notes, but you cannot easily read them. Tools like EditThisCookie act like a magnifying glass, allowing you to see, modify, and remove those notes whenever you want.
