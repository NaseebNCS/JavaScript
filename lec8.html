Filter

In JavaScript, the .filter() method is used to create a new array containing only the elements that pass a specific condition. It's like a "filter" that removes unwanted items and keeps only the ones you want.

How .filter() Works

  1. You call .filter() on an array.
  2. You provide a callback function that defines the condition.
  3. The callback function is applied to each element in the array.
  4. If the condition in the callback function returns true, the element is included in the new array.
  5. If the condition returns false, the element is excluded.

Syntax:

const newArray = originalArray.filter(callbackFunction);
  • originalArray: The array you want to filter.
  • callbackFunction: A function that tests each element.
  • newArray: A new array containing only the elements that passed the test.

Callback Function

The callback function takes three arguments (though you usually only need the first one):

  1. Element: The current element being processed.
  2. Index (optional): The index of the current element.
  3. Array (optional): The original array being filtered.
const numbers = [1, 2, 3, 4, 5];

const evenNumbers = numbers.filter((number) => {
 return number % 2 === 0; // Keep only even numbers
});

console.log(evenNumbers); // Output: [2, 4]

Key Points

  1. Does Not Modify the Original Array: .filter() creates a new array and leaves the original array unchanged.
  2. Returns a New Array: The result is always a new array, even if no elements pass the condition (in which case, it returns an empty array).
  3. Condition-Based: Only elements that satisfy the condition are included in the new array.

Example 1: Filtering Numbers

Filter out numbers greater than 10: 

const numbers = [5, 12, 8, 130, 44];

const filteredNumbers = numbers.filter((number) => {
 return number > 10;
});

console.log(filteredNumbers); // Output: [12, 130, 44]

Example 2: Filtering Strings

Filter out names that start with the letter "A": 

const names = ["Naseeb", "Bob", "Anna", "Lucky", "Ahmed"];

const filteredNames = names.filter((name) => {
   return name.startsWith("A");
});

console.log(filteredNames); // Output: ["Anna", "Ahmed"]

Example 3: Filtering Objects

Filter out users who are older than 18: 

const users = [
 { name: "Naseeb", age: 23 },
 { name: "Bob", age: 17 },
 { name: "Lucky", age: 20 },
];

const adults = users.filter((user) => {
 return user.age > 18;
});

console.log(adults);

Output: [{ name: "Naseeb", age: 23 }, { name: "Lucky", age: 20 }]

Example 4: Filtering Falsy Values

Remove all falsy values (false, 0, "", null, undefined, NaN) from an array: 

const mixedValues = [0, 1, false, true, "", "hello", null, undefined, NaN];

const truthyValues = mixedValues.filter((value) => {
 return value; // Only truthy values are kept
});

console.log(truthyValues); // Output: [1, true, "hello"]

When to Use .filter()

  • When you want to extract specific elements from an array based on a condition.
  • When you want to create a new array without modifying the original array.
  • When working with arrays of objects and need to filter based on object properties.

Function Declarations VS Expressions

Here we are going to examine functions in more detail. We have function declarations and function expressions. What are the differences between them? Let’s see with examples:

Example (Function Declaration):

function walk() {
console.log('walk');
}

This is what we call a function declaration.

There is another method to create a function, called a function expression.

Example (Function Expression): 

let run = function() {
console.log('run');
};

Here, we defined a function. This function doesn’t have a name, and its syntax is very similar to a function declaration, except it doesn’t include a name.

However, we can assign it a name. When we assign a name to this function, it is referred to as a named function expression. If we do not assign it a name, the function is called an anonymous function expression.

One key difference is that in function declarations, we don’t use a terminator (;) at the end of the function block, but in function expressions, we do.

Hoisting

A key difference between function expressions and function declarations is that we can call a declared function before its declaration. 

walk();
function walk() {
 console.log('walk');
}

But we can’t call a function expression in the same way.

run();
const run = function () {
    console.log('run');
};

Here, we will get an error. It is the same as calling a variable before its declaration. 

console.log(n);
let n = "NaseebCodeStudio";

In this case, we will get an error message because we can’t call a variable before its declaration.

When JavaScript executes this code, it moves all the function declarations to the top. This process is called hoisting. Hoisting means first declaration, then assignment.

For variables, hoisting works like this: 

let n = "NaseebCodeStudio";

It moves the declaration part of the variable to the top, like this: 

let n;
n = "NaseebCodeStudio";

However, if we try to access the variable before its declaration, we will get an error or undefined. 

console.log(n); // Error: Cannot access 'n' before initialization

JavaScript breaks the statements into two parts declaration and assignment it moves the declaration at the top and the assignment part is located on the same place.

Arguments

JavaScript is a dynamically typed language. For example, we can set let n = 1; and then change it to n = 'a';. This is perfectly fine because it is one of the characteristics of a dynamically typed language. The same concept applies to the arguments of a function. 

function sum(a, b) {
 return a + b;
}

console.log(sum(1, 2)); // 3

If we pass only one argument:
function sum(a, b) {
 return a + b;
}

console.log(sum(1));

Here, the b parameter is set to undefined. Then, the operation a + b (which means 1 + undefined) results in NaN. Similarly, if we don’t pass any arguments, we don’t get an error. On the other hand, we can also pass additional arguments: 

function sum(a, b) {
 return a + b;
}

console.log(sum(1, 2, 3, 4, 5)); // 3

This is totally fine, but the operation generates 3 because only the first two arguments are used.

What if we want to handle a variable number of arguments? What if we want the flexibility to accept as many arguments as needed to calculate the sum?

Every function in JavaScript has a special object called arguments: 

function sum(a, b) {
 console.log(arguments); // This is the arguments object the function has itself
 return a + b;
}

console.log(sum(1, 2, 3, 4, 5)); // 3

It looks like an array, but it is not an array—it is an object. The keys in the arguments object are the indexes of the arguments passed to the function. The first argument is 1, the second is 2, and so on.

If we want a function to work with a variable number of parameters, we can use a for...of loop with the arguments object inside the function to handle all the arguments as one collection.

function sum(a, b) {
 let total = 0;
 for (let value of arguments) {
 total += value;
 }
 return total;
}

console.log(sum(1, 2, 3, 4, 5, 6)); // 21

Here, we iterate over the arguments object. Earlier, I mentioned that we can use the for...of loop on arrays, but technically, this loop can be used on any object that has an iterator. Plain objects (created using the object literal syntax) don’t have an iterator. However, these particular arguments object has a Symbol.iterator, which means it is iterable, allowing us to use the for...of loop.

We can also remove the a and b parameters because they are not referenced anywhere in the function: 

function sum() {
 let total = 0;
 for (let value of arguments) {
 total += value;
 }
 return total;
}

console.log(sum(1, 2, 3, 4, 5, 6)); // 21

The Rest Operator

In modern JavaScript, if we have a function with a varying number of parameters, we can use the rest operator. Let’s see how it works. We’ll modify the previous function to add parameters to it and prefix them with three dots (...). This is what we call the rest operator. Don’t confuse it with the spread operator—they look exactly the same. Earlier, we used the spread operator with arrays. The spread operator "spreads" an array, meaning it takes its individual elements.

When we use the rest operator with the parameters of a function, we refer to it as the rest operator. 

function sum(...args) {
 console.log(args);
}

console.log(sum(1, 2, 3, 4, 5));

In the result, we can see that args is an array with five elements. This is a real array. If we remove the three dots before args, it means the function will only have a single parameter named args.

Explanation:

When we apply the rest operator to a function’s parameter, we can pass a varying number of arguments. The rest operator collects all the arguments and puts them in an array.

If we want to get the sum of all the numbers in that array, we can use the reduce method. Since args is an array, we can use the .reduce method with two parameters and return the result. 

function sum(...args) {
 return args.reduce((a, b) => a + b);
}

console.log(sum(1, 2, 3, 4, 5, 10)); // Output: 25

In modern JavaScript, we can achieve the same result with less code. Instead of declaring a variable and looping to get the result, we can use a single line of code. This approach is more elegant and professional.

Another Example:

Imagine we want a function to calculate the total price of items in a shopping cart. We also want to apply a discount factor. Let’s pass a parameter for the discount (e.g., 10%), then return the total sum of the prices multiplied by (1 - discount). 

function sum(discount, ...prices) {
 const total = prices.reduce((a, b) => a + b);
 return total * (1 - discount);
}

console.log(sum(0.1, 20, 30)); // Output: 45

Important Note:

You can’t pass any parameters after the rest parameter. Doing so will result in an error. That’s why it’s called the "rest" operator: it collects the rest of the parameters.

Example (Incorrect):

function sum(discount, ...prices, someValue) {
 const total = prices.reduce((a, b) => a + b);
 return total * (1 - discount);
}

console.log(sum(0.1, 20, 30)); // Error

Default Parameters

Sometimes, we want to supply default values to the parameters of a function. For example, consider a function to calculate interest. The formula to calculate interest is: 

function interest(principal, rate, years) {
 return principal * rate / 100 * years;
}

console.log(interest(10000, 3.5, 5)); // Output: 1750

If we want default values for rate and years, we can use logical operators. If rate has a truthy value, we’ll use it; otherwise, a default value of 3.5 is used. Similarly for years. 

function interest(principal, rate, years) {
 rate = rate || 3.5;
 years = years || 5;
 return principal * rate / 100 * years;
}

console.log(interest(10000)); // Output: 1750

ES6 Default Parameters:

With ES6, we can set default values directly in the function parameters.

function interest(principal, rate = 3.5, years = 5) {
 return principal * rate / 100 * years;
}

console.log(interest(10000)); // Output: 1750

Order of Default Parameters

If we set a default value for one parameter, the parameters following it must also have default values. Otherwise, we might get unexpected results.

Example (Incorrect):

function interest(principal, rate = 3.5, years) {
 return principal * rate / 100 * years;
}

console.log(interest(10000)); // Output: NaN

Here, the principal is 10000, rate is 3.5, but years is undefined, so the result is NaN.

If we pass a second parameter, the JavaScript engine might confuse whether it’s for rate or years. 

console.log(interest(10000, 5)); // Still incorrect

To solve this, we can pass undefined for the parameter with a default value so it retains its default value, while other parameters get assigned correctly.

Example (Correct):

console.log(interest(10000, undefined, 5)); // Output: 1750

However, this approach is not very clean.

Higher order of Functions

A Higher-Order Function is a function that takes another function as a parameter or returns a function. 

function greet() {
  return "Hello!";
}

function higherOrder(fn) {
  return fn();
}

console.log(higherOrder(greet));

Function inside a function = Higher-Order Function

Closures

A closure is created when:
An inner function remembers and uses variables of its outer function, even after the outer function has finished. 

function outer() {
  let topic = "JavaScript";

  function inner() {
    console.log(topic);
  }

  return inner;
}

let myFunc = outer();
myFunc();

Explanation:

  • outer() runs and returns inner()
  • Even after outer() is finished,
  • inner() still remembers the variable topic
  • This memory is called a closure

Closure = function + its remembered variables

Lexical Scoping Where variables are accessible based on code position
Closure When a function remembers its outer variables 

function counter() {
  let count = 0;

  return function () {
    count++;
    console.log(count);
  };
}

let increase = counter();
increase(); // 1
increase(); // 2
increase(); // 3

Explanation:

  • count is not destroyed
  • Inner function remembers count
  • This is a closure in action

Getters and Setters

We're going to learn about a special type of method in an object called getters and setters.

Let’s imagine we have a person object with two properties: firstName and lastName. Somewhere in the code, we want to display the full name. 

const person = {
 firstName: 'Naseeb',
 lastName: 'Khan'
};

// We can display the full name using:
console.log(person.firstName + " " + person.lastName);

OR:
console.log(`${person.firstName} ${person.lastName}`);

However, the problem with this approach is that there might be multiple places in our application where we want to display someone’s full name. With the current implementation, we need to repeat this logic multiple times.

A better approach is to define a method in the object, call it fullName, and move the expression using template literals into it. Let’s create another key-value pair and set it to a function: 

const person = {
    firstName: 'Naseeb',
    lastName: 'Khan',
    fullName() {
    return `${this.firstName} ${this.lastName}`;
  }
};

console.log(person.fullName());

This way, we can get the full name by calling person.fullName().

Problems with This Approach:

1. Read-only Access: This method is read-only, meaning we can’t set the fullName property outside of the object. For example, it would be great if we could set the fullName outside the object like this: 

person.fullName = 'Lucky Bhaskar’;

It should automatically update firstName to "Lucky" and lastName to "Bhaskar".

2. Calling it Like a Method: Currently, we need to call fullName() as a method. It would be more intuitive if we could treat it like a property: 

console.log(person.fullName);

Solution: Getters and Setters

Getters allow us to access object properties, and setters allow us to mutate or change them. In this object, we will:

  1. Define a getter for fullName so it can be accessed like a property.
  2. Define a setter for fullName to update firstName and lastName when a new full name is assigned.

Implementation:

const person = {
 firstName: 'Naseeb',
 lastName: 'Khan',
 
 // Getter for fullName
 get fullName() {
 return `${this.firstName} ${this.lastName}`;
 },

 // Setter for fullName
 set fullName(value) {
 const parts = value.split(' ');
 this.firstName = parts[0];
 this.lastName = parts[1];
 }
};

// Using the setter
person.fullName = 'Lucky Bhaskar';

// Output the updated object
console.log(person);

// Output: { firstName: 'Lucky', lastName: 'Bhaskar', fullName: [Getter/Setter] }

Key Points:

Getter Syntax: We use the get keyword to define a getter. The getter allows us to access fullName like a property: 

console.log(person.fullName); // Outputs: Lucky Bhaskar

Setter Syntax: We use the set keyword to define a setter. The setter splits the assigned full name and updates firstName and lastName: 

person.fullName = 'Lucky Bhaskar';
console.log(person.firstName); // Outputs: Lucky
console.log(person.lastName); // Outputs: Bhaskar

By using getters and setters, we improve code reusability and make the code more intuitive to read and use.

Try and Catch

In the previous topic, we assumed that the value passed to a parameter was always a string. But what happens if we pass a Boolean? We get an error stating that split is not a function because split is a method that belongs to strings. Booleans don’t have a split method. Similarly, if we pass null or undefined, we encounter a different error: "Cannot read property 'split' of null."

This situation highlights the need for error handling. When dealing with such cases, it’s essential to validate input values before executing logic. This approach is called defensive programming. By checking the incoming value at the beginning, we can prevent errors and execute our logic safely.

Here’s an example of setting a full name on a person object: 

const person = {
 firstName: 'Naseeb',
 lastName: 'Khan',
 set fullName(value) {
 const parts = value.split(' ');
 this.firstName = parts[0];
 this.lastName = parts[1];
 }
};
person.fullName = 'Lucky Bhasker';

console.log(person);

Now, let's see what happens if we set the full name to null: 

person.fullName = null;
console.log(person);

This throws an error because split cannot be called on null.

Handling Errors

To handle this scenario, we can check the value's type before proceeding: 

const person = {
 firstName: 'Naseeb',
 lastName: 'Khan',
 set fullName(value) {
  if (typeof value !== 'string') return;
 const parts = value.split(' ');
 this.firstName = parts[0];
 this.lastName = parts[1];
 }
};

person.fullName = null;
console.log(person);

Here, the person object retains the original firstName and lastName if the input is invalid.

Throwing Exceptions

Sometimes, we need to notify the application of an issue. This is where exceptions come in. In JavaScript, we can throw an exception using the throw keyword and create an error object.

Here’s how it works:

const person = {
 firstName: 'Naseeb',
 lastName: 'Khan',
 set fullName(value) {
 if (typeof value !== 'string') {
 throw new Error('Value is not a string.');
 }
 const parts = value.split(' ');
 this.firstName = parts[0];
 this.lastName = parts[1];
 }
};

try {
 person.fullName = null;
} catch (e) {
 console.error(e);
}

This approach throws an exception if the input value is not a string.

Handling Empty Strings

What if we pass an empty string? The split method will work, but firstName will become an empty string, and lastName will be undefined. This behavior isn’t desirable.

To handle this, we can add another validation to ensure the input contains both a first and last name: 

const person = {
 firstName: 'Naseeb',
 lastName: 'Khan',
 set fullName(value) {
 if (typeof value !== 'string') {
 throw new Error('Value is not a string.');
 }

 const parts = value.split(' ');
if (parts.length !== 2) {
 throw new Error('Enter a first and last name.');
 }
 this.firstName = parts[0];
 this.lastName = parts[1];
 }
};
try {
 person.fullName = '';
} catch (e) {
 console.error(e);
}
try {
 person.fullName = 'Naseeb'; // Missing last name
} catch (e) {
 console.error(e);
}

Now, an exception is thrown if the input does not meet the required format.

Explanation

By validating input and throwing meaningful exceptions, we ensure the code behaves as expected, even in edge cases. This defensive approach prevents unexpected errors and makes debugging easier.

Local vs Global Scope

In this lecture, we are going to explore the concept of scope, an important topic in JavaScript and programming in general. Scope refers to the context in which variables are declared and determines where these variables can be accessed. In JavaScript, a block is defined by curly braces {}. Variables declared inside a block are not accessible outside of it.

Example 1: Basic Scope

const message = "hello Naseeb";
console.log(message); // Output: hello Naseeb

Here, the message variable is declared in the global scope, so it can be accessed anywhere in the code.

Now, let's declare the same variable inside a block: 

{
 const message = "hello Naseeb";
}
console.log(message); // Output: ReferenceError: message is not defined

In this case, message is only accessible within the block where it is defined. Trying to access it outside the block results in an error

Scope in Functions

Variables declared inside a function are only accessible within that function.

function msg() {
 const message = "hello Naseeb";
}

console.log(message); // Output: ReferenceError: message is not defined

In this case, message is only accessible within the block where it is defined. Trying to access it outside the block results in an error.

Scope in Functions

Variables declared inside a function are only accessible within that function. 

function msg() {
 const message = "hello Naseeb";
}

console.log(message); // Output: ReferenceError: message is not defined

The message constant is accessible only inside the msg function and not outside of it.

Scope in Conditional Blocks

The same concept applies to conditionals such as if statements: 

function msg() {
 const message = "hello Naseeb";
 if (true) {
 const another = "hi Naseeb";
 }

 console.log(another); // Output: ReferenceError: another is not defined
}

Here, another is only accessible within the if block.

Scope in Loops

The same principle applies to loops or any other constructs that use {} braces.

Variable Shadowing

Confusion might arise when we use the same variable name in different scopes: 

function start() {
 const message = "Hello Naseeb";
}

function stop() {
 const message = "Hey Naseeb";
}

This code is valid because the message constants are declared in separate functions. Each message is local to its function, and they do not interfere with each other. This is known as local scope

Global Scope

Now, let's discuss the global scope. Variables declared outside of any function or block are in the global scope and can be accessed anywhere in the code. 

const color = 'red';

function start() {
 const message = "hello Naseeb";
 console.log(color); // Output: red
}
start();

Here, color is accessible inside the start function because it is declared in the global scope.

Local Variables Take Precedence

Local variables or constants take precedence over global ones when they share the same name: 

const color = 'red';

function start() {
 const message = "hello Naseeb";
 const color = "black";
 console.log(color); // Output: black
}

start();

In this example, the local color variable inside the start function overrides the global color variable.

Best Practices

In JavaScript, it is considered bad practice to declare variables in the global scope. Global variables can be accessed and modified from anywhere in the program, which can lead to unexpected bugs and issues.

Think of global variables like a toothbrush—you wouldn't want to share the same toothbrush with everyone. Similarly, each function should have its own variables to avoid conflicts and maintain clean, modular code.

By keeping variables local to the functions or blocks where they are needed, you can reduce the risk of bugs and make your code easier to understand and maintain.

Let vs Var

Before ES6, the only way to declare a variable in JavaScript was by using the var keyword. However, with the introduction of ES6, let and const were introduced to address several issues associated with var.

Problems with var

1. Function Scope: Variables declared with var are function-scoped, meaning they are accessible anywhere within the function they are declared in. This can lead to unintended behavior.

Example with let: 

function start() {
 for (let i = 0; i < 5; i++) {
 console.log(i); // Output: 0, 1, 2, 3, 4
 }
 console.log(i); // Error: i is not defined
}

In this example, the variable i is only accessible within the for loop block. Trying to access it outside the loop will result in an error.

Example with var: 

function start() {
 for (var i = 0; i < 5; i++) {
 console.log(i);
 }
 console.log(i); // Output: 5
}

Here, the variable i is accessible outside the loop because var is function-scoped. Notice that 5 is printed, which is beyond the loop condition (i < 5). This behavior can cause bugs if not handled carefully

2. Lack of Block Scope: Unlike other programming languages, var does not respect block scope. 

function start() {
 for (var i = 0; i < 5; i++) {
 if (true) {
 var color = 'red';
 }
 }
 console.log(color); // Output: red
}

Here, color is accessible outside the if block because var does not have block scope.

Using let:

function start() {
 for (let i = 0; i < 5; i++) {
  if (true) {
 let color = 'red';
 }
 }
 console.log(color); // Error: color is not defined
}

When using let, color is only accessible within the block where it was declared, preventing accidental access outside its intended scope.

3. Global Scope Pollution: Declaring variables with var outside any function attaches them to the global window object in browsers. 

var color = 'red';
let age = 23;

console.log(window.color); // Output: red
console.log(window.age); // Output: undefined

The color variable becomes a property of the window object, which can lead to conflicts if other libraries or scripts use the same variable name. In contrast, let does not attach variables to the window object, reducing the risk of conflicts.

4. Functions and Global Scope: Functions declared in the global scope also become properties of the window object. 

function sayHi() {
 console.log("Hi Naseeb");
}

console.log(window.sayHi); // Output: function sayHi() { console.log("Hi Naseeb"); }
window.sayHi(); // Output: Hi Naseeb

This can cause issues when different scripts define functions with the same name, leading to unexpected overwriting.

The “this” Keyword

The this keyword is one of the features in JavaScript that confuses many developers. The confusion often arises due to poor teaching materials and a lack of clear explanations. Let's clarify how this works with some examples

What is this?

In JavaScript, this refers to the object that is executing the current function. The value of this depends on how the function is called:

  1. 1. When a function is a method inside an object:
    If a function is defined as part of an object (a method), this refers to the object itself.
  2. 2. When a function is a regular function:
    If a function is not attached to any object (i.e., it's a standalone function), this refers to the global object. In browsers, the global object is the window.
  3. 3. When using strict mode ('use strict'):
    In strict mode, this in regular functions will be undefined instead of pointing to the global object.

Example 1: this in an Object Method

const video = {
 title: 'a',
 play() {
 console.log(this);
 }
};

video.play();

Explanation

in this example, play() is a method of the video object. When we call video.play(), this inside the play() method refers to the video object. Therefore, the console will output the video object.

Example 2: Adding Another Method

We can add more methods to the same object, and this will still refer to the video object. 

const video = {
 title: 'a',
 play() {
 console.log(this);
 }
};

video.stop = function() {
 console.log(this);
};

video.stop();

Explanation:

Here, we've added a stop method to the video object. When we call video.stop(), this inside the stop method still refers to the video object. So, the console will log onto the video object again.

Example 3: this in a Regular Function

function showThis() {
 console.log(this);
}

showThis(); // In browsers, this will log the global object (window)

Explanation: Since showThis() is a regular function and not part of any object, this refers to the global object (window in browsers). If you're using strict mode ('use strict'), this will be undefined.

Example 4: Arrow Functions and this

Arrow functions behave differently. They don't have their own this but inherit it from their surrounding (lexical) scope.

const video = {
 title: 'a',
 play() {
   const innerFunction = () => {
    console.log(this);
 };
   innerFunction();
 }
};

Explanation: In this example, innerFunction is an arrow function inside the play method. Arrow functions inherit this from their lexical scope, which in this case is the play method. So, this refers to the video object.

Overview of this Behavior

  • Inside an object method: this refers to the object.
  • In a regular function: this refers to the global object (window), or undefined in strict mode
  • In an arrow function: this inherits from the surrounding scope.
  • When using event listeners: The value of this depends on how the event handler is defined.

Changing the Value of “this”

Hey there! Imagine you have a toy car, and you're the driver. But sometimes, you want your friend to drive the car instead. In JavaScript, "this" is like the driver of the car—it tells who is in control. Let's see how we can change who drives the car using .call(), .apply(), and .bind()

1. Using .call()

.call() lets you call a function and tell it who "this" should be. 

function sayHello() {
 console.log('Hello, my name is ' + this.name);
}

const person1 = { name: 'Naseeb' };
const person2 = { name: 'Sara' };

sayHello.call(person1); // Hello, my name is Naseeb
sayHello.call(person2); // Hello, my name is Sara

Explanation:

  • Think of "sayHello" as a toy car.
  • Using .call(person1) means Ali is driving the car.
  • Using .call(person2) means Sara is driving the car.

2. Using .apply()

.apply() is just like .call(), but instead of giving each argument one by one, you give them in an array. 

function introduce(greeting, age) {
 console.log(greeting + ', my name is ' + this.name + ' and I am ' + age + ' years old.');
}

const person = { name: 'Naseeb' };

introduce.apply(person, ['Hi', 7]); // Hi, my name is Naseeb and I am 7 years old.

It’s like .call(), but you pack your things (arguments) in a bag (array) before giving them

3. Using .bind()

.bind() doesn’t run the function right away. It gives you a new function where "this" is already set. 

function sayGoodbye() {
 console.log('Goodbye from ' + this.name);
}

const person = { name: 'Zara' };

const goodbyeFromZara = sayGoodbye.bind(person);

goodbyeFromZara(); // Goodbye from Zara

.bind() is like giving your friend the keys to your toy car and saying, "You can drive it anytime!"

4. Arrow Functions and "this"

Arrow functions are a little different—they don’t change "this" like regular functions. They use "this" from the place where they were created. 

const person = {
 name: 'Naseeb',
 greet: function() {
 const arrowFunc = () => {
 console.log('Hello, ' + this.name);
 };
 arrowFunc();
 }
};

person.greet(); // Hello, Naseeb

Arrow functions remember who "this" was when they were made. Even though it's inside another function, "this" still points to Bilal.

IIFE

Sometimes, we want to execute a function immediately after defining it. Additionally, functions can encounter issues due to global scope variable pollution. To prevent such issues, we use Immediately Invoked Function Expressions (IIFE). 

Syntax:
(function() {
    // Function body
})();
  • The first set of parentheses () wraps the function definition.
  • The second set () immediately invokes (executes) the function.
  • The semicolon ; at the end ensures that if another IIFE follows, it doesn't cause a syntax error.

Example of a Named IIFE:

(function ncs() {
    console.log("I'm an Immediately Invoked Function Expression (IIFE).");
})();

Example of an Anonymous (No-Name) IIFE:

((name) => {
    console.log(`No-name IIFE function with a single parameter: ${name}`);
})("Naseeb");

This ensures encapsulation and prevents global scope of pollution.
