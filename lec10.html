What’s a Host Environment?

That playground = Host Environment
A host environment is the place where JavaScript runs, and it gives JavaScript extra powers (tools) that are NOT part of core JavaScript.

Example:

JavaScript Alone (Core JS)
By itself, JavaScript can:

  • Do math
  • Store Data
  • Run Logic

But it cannot:

  • Show a webpage
  • Access Buttons
  • Talk to internet
  • Use camera

Now Add Playground (Host Environment)

1. Browser (Like Google Chrome)
When JS runs inside a browser, it gets tools like:

  • document > access HTML
  • window > control browser
  • alert() > show popup

Example:

document.body.style.background = "blue";

💡 This works ONLY in browser, not in pure JS!

2. Server (like Node.js)
When JS runs in Node.js, it gets different tools:

  • fs > read/write files
  • http > create server

Example: 

const fs = require("fs");
fs.writeFileSync("ncs.txt", "Hello!");

This will NOT work in browser!

Key Idea

  • JS itself is small
  • Host environment makes it powerful

Simple Analogy

  • JavaScript = Player
  • Host Environment = Game World

Different worlds > different abilities

Feature Core JS Browser Node.js
Variable ✅ ✅ ✅
DOM (HTML) ❌ ✅ ❌
File System ❌ ❌ ✅
Internet Requests ❌ ✅ ✅

JavaScript is the brain, host environment is the body.

DOM

What is DOM

DOM = Document Object Model
Think of a website like a tree 🌳

  • The whole page = tree
  • Each part (heading, button, image) = node (branch/leaf)

So:Everything you see on a webpage = a DOM node

DOM means: “A way for JavaScript to see and control your HTML”

What is a Node?

Node = any element on the page

Example:

  • <h1> → node
  • <p> → node
  • <button> → node
  • even text → node

Simple: If you can see it (or it exists in HTML), it’s a node.

🚪 Entry Point: document

Before changing anything, we must enter the DOM

// The door is:
document

Think: document = the whole website in JavaScript

🏠 Main Parts of the Page

Your HTML looks like this:

<html>
  <head></head>
  <body></body>
</html>

JavaScript gives you direct access:

1. <html> → Full Page

document.documentElement

This is the entire page

2. <body> → Visible Content

document.body

This is what users actually see:

  • text
  • images
  • buttons

3. <head> → Hidden Setup

document.head

Contain:

  • title
  • links
  • scripts

DOM in Action

DOM allows us to do anything with elements
That means:

1. Children (Direct kids)

Children = direct inside elements

Example:

  • <body> has:
    • <div>
    • <ul>

So: <div> and <ul> are children of <body>

2. Descendants (All kids + grandkids)

Descendants = everything inside (deep)

So <body> has:

  • <div>
  • <ul>
  • <li> inside <ul>
  • <b> inside <li>

Children = Direct
Descendants = All Levels

      console.log("document:", document);

      console.log("body:", document.body);

      console.log("body.children (elements only):", document.body.children);

      // Better when you want only elements:
      console.log("body.firstElementChild:", document.body.firstElementChild);
      console.log("body.lastElementChild:", document.body.lastElementChild);

      // SIBLINGS
      console.log("body.previousSibling (node):", document.body.previousSibling);
      console.log("head.nextSibling (node):", document.head.nextSibling);

      console.log("body.previousElementSibling:", document.body.previousElementSibling); // <head>
      console.log("head.nextElementSibling:", document.head.nextElementSibling);       // <body>

      // PARENT
      console.log("body.parentNode:", document.body.parentNode); // <html>
      console.log("body.parentElement:", document.body.parentElement); // <html>

      // CHECK CHILDREN EXIST
      console.log("body.hasChildNodes():", document.body.hasChildNodes());
    });
  </script>
</body>
</html>
  • childNodes includes text nodes (like "\n " from formatting) and comments.
  • children includes elements only. 
// 3. Why do text nodes appear?
// because formatting like:
<body>
  <div>Begin</div>
</body>

The newline + spaces before <div> becomes a text node.

That’s why:

  • body.childNodes might look like: [#text, div, #text, ul, #text, comment, #text, script]
  • but body.children might look like: [div, ul, script]

4. firstChild vs firstElementChild (common confusion)

  • elem.firstChild → might be a #text
  • elem.firstElementChild → guaranteed to be an element (or null)

So for real-world DOM work, you often want the *Element* versions:

  • firstElementChild
  • lastElementChild
  • nextElementSibling
  • previousElementSibling

DOM Collection

When you get multiple elements like: 

document.body.childNodes

It looks like an array… but ❗
🚫 It is NOT a real array
✅ It is a collection (array-like object)

Simple Meaning
Collection =
Looks like array, but behaves a little different

Why this Matters?

Because:

Feature Works
Loop (for...of) ✅YES
.filter() ❌NO
.map() ❌NO 
// 1. Looping works 
for (let node of document.body.childNodes) {
  console.log(node);
}

This Prints:

  • text nodes
  • elements
  • everything inside <body>

Why it works: Because collection is iterable

// 2. Array Methods don't work
console.log(document.body.childNodes.filter);

// output: undefined

Because: it is not a real array.

Solution: convert to real Array

let nodes = Array.from(document.body.childNodes);

nodes.filter(node => node.nodeType === 1);

Now .filter() works

Simple Analogy:
Collection = Fake Juice box
Array = Real Juice
Fake looks same... but not same inside.

  document.body.parentNode === document.documentElement

  <p>Paragraph</p>

// Get all nodes
let nodes = document.body.childNodes;

// Loop through them
for (let node of nodes) {
  console.log(node);
}

// Convert to array
let arr = Array.from(nodes);

// Filter only ELEMENTS
let elementsOnly = arr.filter(node => node.nodeType === 1);

console.log(elementsOnly);

Access Textual Content

There are three ways to access an HTML element's text or content.

  • innerHTML
  • innerText
  • textContent

'Text' refers only to the plain text, whereas 'content' refers to everything inside the element, which can include both text and other HTML elements.

Comparison

Property Reads HTML? Shows Hidden text? Affects Styling?
innerHTML ✅YES ✅YES ❌NO
innerText ❌NO ❌NO ✅YES
textContent ❌NO ✅YES ❌NO 
<div id="box">
  Hello <b>Naseeb</b>
  <span style="display:none">Hidden</span>
</div>

// JavaScript 
// 1. innerHTML
console.log(document.getElementById("box").innerHTML);

// Output: Hello <b>Naseeb</b> <span style="display:none">Hidden</span>

Meaning: it returns everything Including HTML tags, innerHTML says give me Exact HTML inside. 

// Changing the content of a box element
document.getElementById("box").innerHTML = "<h1>NaseebCodeStudio</h1>";

It will actually create an <h1> element

// 2. innerText
console.log(document.getElementById("box").innerText);

// Output: Hello Naseeb

Meaning: it returns only visible text,
Hiddent text is ignored
HTML tags removed
innerHTML says Give me What user sees. 

// 3. textContent
console.log(document.getElementById("box").textContent);

// Output: Hello Naseeb Hidden

textContent: it returns all text
includes hiddent text
No HTML tags
textContent says Give me all text, I don't care if hidden.

Easy Analogy

Imagine a Box:
Inside:

  • Text
  • Hidden text
  • Some HTML styling
Property What it Does
innerHTML Shows full box with structure
innerText Shows only what you see
textContent Shows All text (even Hidden) 
// Practical Examples

// 1. Change Text Only
document.getElementById("box").innerText = "Hello User!";
// safe, no HTML added 

// 2. Add HTML
document.getElementById("box").innerHTML = "<b>Bold Text</b>";
// Creates real <b> tag

// 3. Fast Text Change
document.getElementById("box").textContent = "Fast update!";
// Faster than innerText

Important Difference (Real World)

Speed:

  • textContent is fastest
  • innerText is slower (because checks styling)

Styling Effect

element.innerText
// depends on CSS (visible or not)

element.textContent
// ignores CSS completely

Security Warning

innerHTML
// can be dangerious if you insert user data 

// Example:
elemenet.innerHTML = userInput;

Can allow XSS attack

Engineer Tip

In real projects: prefer to use textContent
Avoid too much:
innerHTML (unless really needed)
