Lexical vs Dynamic Scoping

What is Scope?

Scope = Where a variable can be used.

Example:

let name = "Naseeb";

function greet() {
    console.log(name);
}

greet();

// Output: Naseeb

Why: Because name is visible inside greet().

Real Life Example

Imagine your home.

  • Bedroom = one scope
  • Kitchen = another scope
  • Hall = another scope

If you keep your phone in the hall, everyone in the house can access it.
If you keep money inside your bedroom drawer, only you can access it.
Variables work exactly like this.

There are Two Types of Scoping

  1. Lexical Scope
  2. Dynamic Scope

What does Lexical mean?

Lexical Means: Where the function is written in the code. Same as where are you physically

JavaScript uses Lexical Scope.
It does NOT use Dynamic Scope.

Lexical Scope (JavaScript's Real Behavior)

Example 1

let fruit = "Mango";

function showFruit() {
    console.log(fruit);
}

showFruit();

// Output: Mango

showFruit() looks outside itself and finds fruit.

Example 2

let x = 10;

function first() {
    console.log(x);
}

function second() {
    let x = 20;
    first();
}

second();

// Output: 10

Many beginners think output should be 20.

Not!
Let's understand.

Memory Picture:

Global Scope
│
├── x = 10
│
├── first()
│
└── second()
     │
     └── x = 20

Inside second(), we call first().
But first() was CREATED in the global scope.
So when it needs x, it goes to where it was born (global scope).

It finds:

x = 10

// Therefore Output: 10

Golden Rule of Lexical Scope

A function remembers where it was created, NOT where it was called.

Remember this line forever.

Another Example:

let city = "Quetta";

function outer() {

    function inner() {
        console.log(city);
    }

    inner();
}

outer();

// Output: Quetta

Example 3:

// Lexical Scope
function nk1(){
  let nk = "Naseeb";
  
  function nk2(){
    console.log(nk); // Naseeb
  }
}

// Dynamic Scope
let nk = "Naseeb Khan";

function newNk1(){
    console.log(nk); // Naseeb Khan
}

function newNk2(){
  let nk = "Lucky Khan";

  newNk1();
}

newNk2(); // Naseeb Khan why because JS has lexical if it would have a dynamic scope we would get Lucky Khan

Nested Functions

function parent() {

    let money = 500;

    function child() {
        console.log(money);
    }

    child();
}

parent();

// Output: 500

Why: Because child function can see variables of its parent.

Scope Chain

Global Scope
    ↓
Parent Scope
    ↓
Child Scope

Child first checks:

  1. Inside itself
  2. Parent
  3. Global

If not Found: ReferenceError

Example:

function parent() {

    let age = 25;

    function child() {
        console.log(age);
    }

    child();
}

parent();

// Output: 25

Child borrows variables from parent.

What is Dynamic Scope?

Dynamic scope means: variable is found according to WHO called the function.

This is NOT how JavaScript Works.

Suppose JS had dynamic scope.

let x = 10;

function first() {
    console.log(x);
}

function second() {
    let x = 20;
    first();
}

second();

Under Dynamic Scope:

Output would be: 20

Because first() was called by second().

But JS says: "No, I don't care who called me. I care where i was created.

Therefore output is: 10

Difference Between Lexical and Dynamic Scope

Lexical Scope Dynamic Scope
Used by JavaScript Not used by JavaScript
Depends on where function is written Depends on who called function
Fixed at creation time Changes at runtime
predictable Can Change unexpectedly
Easier to understand More confusing

Real Life ANALOGY

Lexical Scope:
Imagine your home address.
You were born in Quetta.
Even if your friend from Karachi invites you, your home address is still Quetta.
Functions Behaves like this.
They remember where they came from.

Dynamic Scope:
Imagine changing your home address every time someone calls you.
Nk Calls > You live in Lahore.
Ak calls > You live in Karachi.
Naseeb calls > You live in Quetta.
That's why JavaScript uses Lexical Scope.

Closure Comes From Lexical Scope

function outer() {

    let count = 0;

    function inner() {
        count++;
        console.log(count);
    }

    return inner;
}

let myFunction = outer();

myFunction();
myFunction();
myFunction();

// Output: 
// 1
// 2
// 3

Why: Because inner() remembers the place where it was created.
This memory is called Closure.
Closures are possible because JavaScript uses Lexical Scope.

Rules

Rule 1: Scope means where variables are accessible.

Rule 2: JavaScript uses Lexical Scope.

Rule 3: A function remembers where it was created, not where it was called.

Rule 4: Child functions can access parent variables.

Rule 5: Dynamic Scope depends on who called the function but JavaScript does not use it.

Final Example:

let name = "Global";

function show() {
    console.log(name);
}

function test() {
    let name = "Local";
    show();
}

test();

Question? What will be Printed?

Answer: Global

Because: show() remembers where it was Created (global scope), not where it was Called (Inside test).

One-Line Memory Trick

Lexical Scope = Birth Place of Function

Dynamic Scope = Caller of Function

JS Always chooses Birth place.

Synchronous vs Asynchronous JavaScript

This is one of the most important concepts in JavaScript.
If you understand this topic properly, then understanding Callbacks, Promises, Async/Await, Fetch API, Event Loop becomes much easier.

What is Synchronous JavaScript?

The word synchronous means:
👉 Do one task at a time.
👉 Wait for the current task to finish before starting the next task.
JavaScript executes code line by line.
Think of a single cashier at a shop.

Real Life Example

You go to a cashier.

  1. First customer pays.
  2. Then second customer pays.
  3. Then third customer pays.

Nobody can jump ahead.

Everything happens in order.

Synchronous Code Example

console.log("Step 1");

console.log("Step 2");

console.log("Step 3");

// Output
Step 1
Step 2
Step 3

JavaScript reads:

Line 1 → Execute
Line 2 → Execute
Line 3 → Execute

Simple.

Problem with Synchronous Programming

Imagine a website needs data from a server.
The server takes 5 seconds.
If JavaScript waits for those 5 seconds, the whole website freezes.

Example

console.log("Start");

waitFor5Seconds();

console.log("End");

// Output: 
Start
(wait 5 seconds)
End

User cannot interact during that wait.
This creates a bad user experience.

What is Asynchronous JavaScript?

Asynchronous means:
👉 Don't wait.
👉 Start a task.
👉 Continue other work.
👉 Come back when the task finishes.

Real Life Example

Suppose you order food in a restaurant.

// Synchronous Way
Order food
Sit and do absolutely nothing
Food arrives

Waste of time.

// Asynchronous Way
Order food
Talk with friends
Check mobile
Food arrives
Eat

You continue doing other work.
This is exactly how asynchronous JavaScript works.

Asynchronous Code Example

console.log("Start");

setTimeout(() => {
  console.log("Data Received");
}, 3000);

console.log("End");

Predict the Output

Many Beginners think: 

Start
(wait 3 seconds)
Data Received
End

// Wrong ❌

// Actual Output: 
Start
End
Data Received

Why? Because JavaScript doesn't wait for setTimeout.
It schedules it and moves on.

Callback Pattern and Callback Hell in JavaScript

Before learning Promises and Async/Await, you must understand callbacks because they were the original way JavaScript handled asynchronous operations.

What is a Callback?

A callback is simply: A function passed as an argument to another function. The function is called later when needed.
Think of it like giving your phone number to a delivery rider.
You tell him: "When you arrive, call me."
You don't keep waiting at the gate. You continue your work.
The rider's call is like a callback function.

Simple Callback Example

function greet(name, callback) {
  console.log("Hello " + name);

  callback();
}

function sayBye() {
  console.log("Good Bye");
}

greet("Naseeb", sayBye);

// Output:
Hello Naseeb
Good Bye

Why Callbacks Are Useful

Imagine a website is loading user data.
Fetching data takes time.
We don't know exactly when the server will respond.
So we say: 

When data arrives,
run this function.

That function is the callback.

Practical Example

setTimeout(() => {
  console.log("Data Loaded");
}, 3000);

// Output after 3 seconds: Data Loaded

The arrow function is a callback.

JavaScript says: Run this function later.

Real Website Example

Imagine a login system.

function loginUser(callback) {
  setTimeout(() => {
    console.log("User Logged In");

    callback();
  }, 2000);
}

loginUser(() => {
  console.log("Redirecting to Dashboard");
});

// Output: 
User Logged In
Redirecting to Dashboard

Real Lefe Flow:

User clicks Login
↓
Server verifies account
↓
Login successful
↓
Redirect dashboard

The redirect function is the callback.

Imagine an e-commerce website.

// FLOW

Get User
↓
Get Orders
↓
Get Order Details
↓
Process Payment
↓
Send Email

Each step depends on the previous one.

What is Callback Hell?

Callback Hell happens when callbacks become deeply nested.
Code starts moving to the right side again and again.
It becomes hard to read, debug, and maintain.

Example of Callback Hell

getUser(function(user) {

  getOrders(user, function(orders) {

    getOrderDetails(orders, function(details) {

      processPayment(details, function(payment) {

        sendEmail(payment, function() {

          console.log("Done");

        });

      });

    });

  });

});

Look at the indentation.
This shape is often called: 

  Pyramid of Doom

Because the code keeps shifting right.

Why is This Bad?

1. Hard to Read

Finding logic becomes difficult. 

getUser(() => {
  getOrders(() => {
    getOrderDetails(() => {
      processPayment(() => {
        sendEmail(() => {

        });
      });
    });
  });
});

2. Hard to Debug
If something fails: 

processPayment()

You must trace through multiple levels.

3. Hard to Maintain

Adding another step means another nested callback.

getUser(() => {
  getOrders(() => {
    getDetails(() => {
      processPayment(() => {
        sendEmail(() => {
          updateInventory(() => {

          });
        });
      });
    });
  });
});

Now it's even worse.

Real Website Example

Imagine a food delivery app.

// FLOW

Find User Location
↓
Find Nearby Restaurants
↓
Load Menu
↓
Place Order
↓
Process Payment
↓
Send Confirmation

Old callback approach:

getLocation(function(location) {

  getRestaurants(location, function(restaurants) {

    getMenu(restaurants, function(menu) {

      placeOrder(menu, function(order) {

        processPayment(order, function(payment) {

          sendConfirmation(payment, function() {

            console.log("Order Complete");

          });

        });

      });

    });

  });

});

Very difficult to manage.

How JS solved callback hell

JavaScript introduced:

  1. Promises
  2. Async/Await

Instead of:

getUser(function(user) {

});

we can write:

getUser()
  .then(getOrders)
  .then(getOrderDetails)
  .then(processPayment)
  .then(sendEmail);

Much cleaner.

And with Async/Await:

const user = await getUser();
const orders = await getOrders(user);
const details = await getOrderDetails(orders);
const payment = await processPayment(details);

await sendEmail(payment);

Looks almost like normal synchronous code while still being asynchronous.

Promises in JavaScript

Today we are learning one of the most important JavaScript concepts.
Before Promises, developers used callbacks.

The problem was:

getUser(() => {
  getOrders(() => {
    getDetails(() => {
      processPayment(() => {
        sendEmail(() => {

        });
      });
    });
  });
});

This became Callback Hell.
To solve this problem, JavaScript introduced Promises.

What is a Promise?

A Promise is an object that represents the result of a future operation.
Simple: 👉 A Promise is a guarantee that something will happen later.
Think about ordering food online.
When you place an order: 

Order Placed
↓
Restaurant Preparing Food
↓
Food Delivered OR Delivery Failed

You don't get the result immediately.
You wait.
This is exactly how a Promise works.

Real Life Exampleh4>

Imagine your friend says: "I promise I'll return your book tomorrow."
Now there are only 2 possibilities:

Success: Friend returns book

Promise fulfilled.

Failure: Friend loses book

Promise rejected.
JavaScript Promises work exactly the same way.

Promise States

A promise can be in 3 states.

1. Pending

Task is still running.

Food is being prepared.

2. Fulfilled (Resolved)

Task completed successfully.

Food delivered

3. Rejected

TAsk Failed.

Delivery Failed

Creating a Promise

Syntax:

const promise = new Promise((resolve, reject) => {

});

What is resolve?

Used when everything succeeds.

resolve("Success");

// Means: 
Task completed successfully

What is Rejected?

Used when something fails:

reject("Failed");

// Means: 
Task Failed

First Promise Example

const myPromise = new Promise((resolve, reject) => {
  resolve("Data Loaded Successfully");
});

// Here: 
resolve(...)

runs immediately.
So the Promise becomes:Fulfilled

Reading the Result with .then()

To receive successful data:

myPromise.then((result) => {
  console.log(result);
});

// Output: 
Data Loadedd Successfully

Example: 

const myPromise = new Promise((resolve, reject) => {
  resolve("Data Loaded Successfully");
});

myPromise.then((result) => {
  console.log(result);
});

// Output: 
Data Loaded Successfully

Understanding .then()

Think of .then() as saying: "When the Promise succeeds, do this."
Example:

promise.then(() => {

});

// Meaning: 
if successful, 
run this code.

Reject Example

Let's create a failure.

const myPromise = new Promise((resolve, reject) => {
  reject("Server Error");
});

// Promise State: Rejected

Handling Errors with .catch()

To handle failure: 

myPromise.catch((error) => {
  console.log(error);
});

// Output: Server Error

Example:

const myPromise = new Promise((resolve, reject) => {
  reject("Server Error");
});

myPromise.catch((error) => {
  console.log(error);
});

// Output: Server Error

Real Website

Suppose a website loads products.

const productsPromise = new Promise((resolve, reject) => {

  const productsLoaded = true;

  if (productsLoaded) {
    resolve("Products Loaded");
  } else {
    reject("Failed To Load Products");
  }

});

// Success Handling: 
productsPromise.then((message) => {
  console.log(message);
});

// Output: Products Loaded 

// Failure Handling:
productsPromise.catch((error) => {
  console.log(error);
});

// Output: Failed to Load Products

Using Then and Catch Together

This is the most common pattern in real projects.

const promise = new Promise((resolve, reject) => {

  const success = true;

  if (success) {
    resolve("User Found");
  } else {
    reject("User Not Found");
  }

});

promise
  .then((result) => {
    console.log(result);
  })
  .catch((error) => {
    console.log(error);
  });

// Output: User Found

Promise Chaining

One promise can pass data to another .then()

Promise.resolve(10)
  .then((num) => {
    return num * 2;
  })
  .then((num) => {
    return num * 3;
  })
  .then((num) => {
    console.log(num);
  });

// Output: 60

Now you have learned:

Synchronous vs Asynchronous
↓
Callbacks

Callback Hell

Now let's learn the modern way used in almost every professional JavaScript application: Async/Await

Why Do We Need Async/Await?

Look at this pormise code:

fetch("/users")
  .then((response) => response.json())
  .then((data) => {
    console.log(data);
  })
  .catch((error) => {
    console.log(error);
  });

Nothing wrong with it:

But Imagine:

getUser()
  .then(getOrders)
  .then(getOrderDetails)
  .then(processPayment)
  .then(sendEmail)
  .then(updateInventory)
  .then(generateInvoice)
  .catch(handleError);

As applications grow, chains become harder to read.

JavaScript introduced Async/Await to make asynchronous code look like normal code.

What is async?

The async keyword is used before a function.

async function myFunction() {

}

This tells JavaScript: "This function will work with Promises."

Example:

async function greet() {
  return "Hello";
}

Looks like a normal return.

But JavaScript secretly converts it into: 

Promise.resolve("Hello");

Proof:

async function greet() {
  return "Hello";
}

greet().then((data) => {
  console.log(data);
});

// Output: 
Hello

What is await?

await means: Wait for a Promise to finish.
Get its result. Think of a restaurant.

// Without await:

Order food
Keep doing work
Food arrives later

// With await:

Order food
Wait until food arrives
Then continue

First Await Example

function getData() {
  return Promise.resolve("User Data");
}

// Using Promise style:
getData().then((data) => {
  console.log(data);
});

// OutPut: User Data


// 🚀 Now with await:
async function showData() {
  const data = await getData();

  console.log(data);
}

showData();

// Output: User Data

Much cleaner.

How Await Works

const data = await getData();


// JavaScript says:

Pause this async function
↓
Wait for Promise
↓
Get result
↓
Continue execution

Real Website Example

Suppose a website loads user information.

Promise style:

fetch("/user")
  .then((response) => response.json())
  .then((data) => {
    console.log(data);
  });

Async/Await style:

async function loadUser() {
  const response = await fetch("/user");

  const data = await response.json();

  console.log(data);
}

Most developers find this easier to read.

Multiple Async Operations

Imagine an e-commerce website.

// Flow: 

Get User
↓

// Using Async/Await:

  const user = await getUser();

This looks almost like normal synchronous code.
But behind the scenes it is asynchronous.

Handling Errors with Try/Catch

With Promises:

fetch("/user")
  .then(...)
  .catch(...)

With Async/Await:

try {

}
catch(error) {

}

Example:

async function loadUser() {

  try {

    const response = await fetch("/user");

    const data = await response.json();

    console.log(data);

  } catch (error) {

    console.log("Something went wrong");

  }

}

Real Login Example

Imagine a login page:

async function login() {

  try {

    const response = await fetch("/login");

    const user = await response.json();

    console.log("Welcome", user.name);

  } catch (error) {

    console.log("Login Failed");

  }

}

Real websites use this pattern everywhere.

Real Production Example

Suppose a shopping website.
When user opens product page: 

async function loadPage() {

  try {

    const product = await getProduct();

    const reviews = await getReviews();

    const seller = await getSeller();

    displayPage(product, reviews, seller);

    showError();

  }

Promise vs Async/Await

// Promise Style

getUser()
  .then((user) => {
    return getOrders(user);
  })
  .then((orders) => {
    return getProducts(orders);
  })
  .catch((error) => {
    console.log(error);
  });

try {

}

Most developers prefer the second version.
